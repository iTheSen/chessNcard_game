# 纸牌项目程序设计与扩展指南（cocos2d-x 3.17）

## 1. 文档目标

本文档用于交付“可维护、可扩展”的程序设计说明，重点回答两件事：

1. 现有代码结构如何支撑持续迭代。
2. 在当前架构下，未来如何低风险新增一个卡牌和一个新类型回退功能。

该文档可作为后续需求评审、任务拆分、开发交接的基线。

---

## 2. 架构分层与职责边界

当前实现遵循需求文档建议的分层：

- `Classes/configs/`：静态配置（关卡、资源映射）。
- `Classes/models/`：运行时数据模型（卡牌、游戏态、回退记录）。
- `Classes/views/`：纯展示与输入捕获，不承载业务规则。
- `Classes/controllers/`：业务编排层，连接 View 与 Model。
- `Classes/managers/`：有状态管理器（如回退栈）。
- `Classes/services/`：无状态服务（规则判断、模型转换）。
- `Classes/utils/`：通用工具。
- `Classes/scenes/`：场景入口。

### 2.1 设计原则（当前版本已落地）

- 规则逻辑集中在 `services/` 与 `controllers/`，避免散落到 `views/`。
- 运行时状态统一归 `models/`，避免多处复制状态。
- 回退统一通过 `UndoManager` 管理，避免绕开回退栈。
- 新功能优先“增量扩展”，减少横向侵入。

## 3. 当前核心流程

### 3.1 启动流程

`main -> AppDelegate -> GameScene -> GameView + GameController -> level_1.json`

关键文件：
- `proj.ios_mac/mac/main.cpp`
- `Classes/AppDelegate.cpp`
- `Classes/scenes/GameScene.cpp`

### 3.2 点击匹配流程

1. `CardView` 捕获点击并回调 `GameView`。
2. `GameView` 透传给 `GameController`。
3. `GameController` 路由到 `PlayFieldController` 或 `StackController`。
4. 控制器更新 `GameModel`，生成 `UndoModel`。
5. 触发 `MoveTo` 动画，动画完成后刷新视图状态。

关键文件：
- `Classes/views/CardView.cpp`
- `Classes/controllers/GameController.cpp`
- `Classes/controllers/PlayFieldController.cpp`
- `Classes/controllers/StackController.cpp`

### 3.3 回退流程

1. 点击回退按钮。
2. `UndoManager::popRecord` 弹出最近记录。
3. `GameController::applyUndoRecord` 恢复模型。
4. 卡牌反向 `MoveTo`，再统一刷新视图。

关键文件：
- `Classes/managers/UndoManager.cpp`
- `Classes/models/UndoModel.h/.cpp`
- `Classes/controllers/GameController.cpp`

## 4. 数据模型稳定性约束

### 4.1 `GameModel` 关键约束

- `trayTopCardId` 必须始终指向 `CZT_TRAY` 区域牌。
- `nextStackIndex` 必须与 `stackCardIds` 同步。
- 任何控制器动作结束后，`CardModel.zone` 与 `CardModel.currentPosition` 必须一致。

### 4.2 `UndoModel` 关键约束

- 必须能完整恢复“动作前状态”，至少包含：
- 被移动卡牌 ID
- 来源区域
- 来源坐标
- 操作前顶部手牌 ID
- 操作前堆牌索引

## 5. 扩展能力评估（承接需求量的关键）

当前结构对新增需求的承接能力来自以下设计：

1. 配置与运行态分离：
`LevelConfigLoader + GameModelFromLevelGenerator` 使关卡变更主要落在 JSON。

2. 规则与显示分离：
`GameRuleService` 负责规则判断，`View` 只负责渲染和输入。

3. 回退统一建模：
`UndoModel + UndoManager + applyUndoRecord` 形成可持续扩展的回滚骨架。

4. 主控制器编排：
`GameController` 集中处理流程，便于插入新功能节点（如提示、自动匹配、道具）。

## 6. 扩展示例 A：如何新增一个卡牌

这里给出两种常见场景。

### 6.1 场景 A1：新增“普通卡牌实例”（不新增牌型）

目标：只是在某关新增一张已有点数/花色卡牌。

步骤：
1. 修改关卡配置：`Resources/configs/levels/level_x.json`。
2. 在 `Playfield` 或 `Stack` 数组增加一项：`CardFace`、`CardSuit`、`Position`。
3. 运行后由 `LevelConfigLoader` 自动加载。
4. `GameModelFromLevelGenerator` 自动生成 `CardModel`。
5. `GameView::buildCards` 自动创建并展示 `CardView`。

涉及文件：
- `Resources/configs/levels/level_x.json`

通常不需要改动：
- `models/`、`controllers/`、`services/`

### 6.2 场景 A2：新增“新牌型”（例如 Joker）

目标：新增一种当前枚举中不存在的牌型，并参与显示/规则。

步骤：
1. 扩展枚举：`Classes/models/CardTypes.h` 中 `CardFaceType`。
2. 扩展资源映射：`Classes/configs/models/CardResConfig.cpp` 的 `faceToToken`。
3. 补齐资源文件：`Resources/res/cards/number/` 与必要花色图。
4. 更新牌面拼装逻辑（如需要）：`Classes/services/CardSpriteComposerService.cpp`。
5. 更新规则判断：`Classes/services/GameRuleService.cpp`（若 Joker 匹配规则特殊）。
6. 在关卡 JSON 中配置新牌验证。

涉及文件：
- `Classes/models/CardTypes.h`
- `Classes/configs/models/CardResConfig.cpp`
- `Classes/services/CardSpriteComposerService.cpp`
- `Classes/services/GameRuleService.cpp`
- `Resources/res/cards/...`
- `Resources/configs/levels/level_x.json`

### 6.3 新增卡牌验收清单

- 牌面是否正常显示（大小图、花色、翻转）。
- 点击交互是否与覆盖规则一致。
- 匹配规则是否符合预期。
- 回退后该牌状态是否可恢复。

## 7. 扩展示例 B：如何新增一种“回退类型”

示例目标：新增一种新动作 `UAT_HINT_AUTO_MATCH`（由提示功能触发的自动匹配），并支持回退。

### 7.1 变更步骤

1. 扩展动作枚举
- 文件：`Classes/models/UndoModel.h`
- 在 `UndoActionType` 增加 `UAT_HINT_AUTO_MATCH`。

2. 评估并补充回退数据字段
- 文件：`Classes/models/UndoModel.h/.cpp`
- 如果现有字段不足以恢复，新增字段并补齐 `toJson/fromJson`。

3. 在动作发生处写入回退记录
- 文件：对应业务控制器（通常 `GameController` 或新 Controller）
- 记录动作前的关键状态（顶部牌、堆索引、来源坐标等）。

4. 在回退应用入口增加处理分支
- 文件：`Classes/controllers/GameController.cpp`
- 函数：`applyUndoRecord(...)`
- 根据新 `UndoActionType` 恢复模型。

5. 刷新视图并验证动画
- 保持“先恢复模型，再刷新视图，再状态文案”顺序。

### 7.2 建议的分发写法（伪代码）

```cpp
void GameController::applyUndoRecord(const UndoModel& undoRecord)
{
    switch (undoRecord.getActionType()) {
        case UAT_STACK_TO_TRAY:
        case UAT_PLAYFIELD_TO_TRAY:
            // 现有逻辑
            break;
        case UAT_HINT_AUTO_MATCH:
            // 新增恢复逻辑
            // 1) 恢复卡牌 zone/position
            // 2) 恢复 trayTopCardId
            // 3) 恢复 nextStackIndex（如有变更）
            break;
        default:
            break;
    }
}
```

### 7.3 新回退类型验收清单

- 连续多步回退是否保持后进先出。
- 新动作与旧动作混合后回退是否正确。
- 回退后交互可操作牌集合是否正确刷新。
- 状态栏文案是否准确。

## 8. 影响面矩阵（任务评审用）

### 8.1 新增普通卡牌实例

- 高概率改动：`Resources/configs/levels/*.json`
- 中低概率改动：无
- 风险等级：低

### 8.2 新增新牌型

- 高概率改动：`CardTypes`、`CardResConfig`、资源目录、规则服务
- 中概率改动：牌面拼装服务
- 风险等级：中

### 8.3 新增回退类型

- 高概率改动：`UndoModel`、`GameController::applyUndoRecord`、触发动作控制器
- 中概率改动：状态栏文案与动画回调
- 风险等级：中高（涉及状态一致性）

---

## 9. 交付建议（提高可承接需求量）

1. 每个新需求必须附带“影响面矩阵 + 验收清单”。
2. 规则变更只改 `services/` 与 `controllers/`，避免进入 `views/`。
3. 新增回退功能时先扩 `UndoModel`，再落业务动作，最后补回退分支。
4. 每个模块改动后至少完成一次“混合操作 + 多步回退”手测。
